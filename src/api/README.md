# src/api — WebSocket handler: логика во время игры

## Общая схема

```
Клиент A          Клиент B
   |                 |
   | WS /ws          | WS /ws
   |                 |
   v                 v
WebsocketsHandler (один на соединение)
   |
   +-- init_user_id_()      <- аутентификация по ПЕРВОМУ сообщению
   |
   +-- read_loop_()         <- читает действия от клиента
   |       |
   |       v
   |   GameRoom::send_message()   <- маршрутизация действий (TODO: receive_message)
   |
   +-- send_loop_()  [async]      <- ждёт ответов и отправляет клиенту
           |
           v
       PlayerSession::pop_wait()  <- блокируется до появления сообщения
```

---

## Шаг 1 — Подключение и аутентификация (`init_user_id_`)

После установки WebSocket-соединения клиент **обязан** первым же сообщением отправить:

```json
{ "token": "<raw_token>", "game_id": 42 }
```

Сервер:

1. Вычисляет `jti = SHA256(token)` и делает SELECT в `auth_tokens` → получает `user_id`.
2. Ищет `GameRoom` с нужным `game_id` в `StorageClient` (in-memory хранилище).
3. Проверяет, что пользователь уже присоединился к игре через HTTP (`GameRoom::check_for_user`).

При любой ошибке — закрывает соединение с кодом `kBadMessageData`.

---

## Шаг 2 — Два параллельных цикла

После успешной аутентификации `Handle` запускает два корутинных цикла:

### `read_loop_` (основной поток)

Читает входящие сообщения от клиента через `chat.TryRecv()`. Использует поллинг с паузой 100 мс, если сообщений нет. При получении `close_status` — закрывает соединение и завершает цикл.

Входящие сообщения передаются в `GameRoom::send_message()`.

### `send_loop_` (асинхронный)

Блокируется на `PlayerSession::pop_wait()` — ждёт появления сообщения в очереди сессии игрока. Как только сообщение появляется — сразу отправляет его клиенту через `chat.Send()`.

---

## Шаг 3 — Обработка действий игрока (`GameRoom`)

`GameRoom::receive_message()` парсит JSON и маршрутизирует по полю `"action"`:

| Значение `action` | Метод           | Описание |
|-------------------|-----------------|----------|
| `"place"`         | `action_place`  | Расставить тайлы на доске |
| `"submit"`        | `action_submit` | Подтвердить слово |
| `"change"`        | `action_change` | Обменять тайлы |
| `"pass"`          | `action_pass`   | Пропустить ход |
| `"end"`           | `action_end`    | Завершить игру |
| `"state"`         | `game_state_for_user_` | Запросить текущее состояние |

После обработки действия результат кладётся в очередь нужного игрока через `PlayerSession::send_raw_message()`, откуда его заберёт `send_loop_` и отправит клиенту.

---

## Очередь сообщений (`PlayerSession`)

Каждый игрок имеет свою `PlayerSession` — изолированную очередь сообщений:

```
send_raw_message(msg)  →  queue_.push(msg) + cv_.NotifyOne()
pop_wait()             →  cv_.Wait(...) + queue_.pop()
Close()                →  closed_ = true + cv_.NotifyAll()   <- разблокирует send_loop_
```

`Close()` — предусмотрен механизм чистого завершения: при вызове `pop_wait` возвращает пустую строку, что позволяет `send_loop_` завершиться. Однако `Close()` сейчас нигде не вызывается.

---

## Известные проблемы

**Не все Действия игроков реализованы.** Метод `action_end` объявлен, но нет реализации `GameRoom.cpp`.

**`send_loop_` не завершается при дисконнекте.** После закрытия соединения клиентом `send_loop_` продолжает висеть заблокированным на `pop_wait`. Нужно вызывать `session->Close()` при завершении `read_loop_`.
